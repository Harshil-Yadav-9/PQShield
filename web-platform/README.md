This is a [Next.js](https://nextjs.org) project bootstrapped with [`create-next-app`](https://nextjs.org/docs/app/api-reference/cli/create-next-app).

## How it fits together

- **Frontend** — this Next.js app. Runs locally with `npm run dev`; deploys to Vercel.
- **Scanner service** — `pqc_scanner/service.py`, a FastAPI wrapper around the TLS/PQC scanner. Runs locally with `uvicorn`; deploys as a container (Dockerfile at repo root) to Railway/Fly/Render — it needs raw sockets and can run 60–90s per request, so it can't be a Vercel serverless function. It exposes `POST /scan` (returns `{ report, raw }` — the web-shaped report plus the raw CBOM JSON) and `POST /export/excel` (rebuilds the Excel risk report from previously-scanned raw JSON, no re-scan).
- **Database** — Postgres on [Neon](https://neon.tech), via Prisma 7 (`@prisma/adapter-neon`, Neon's serverless driver — works identically on `npm run dev` and on Vercel's serverless functions, no connection-pooling setup needed). Stores `User`/`Account`/`Session` (Auth.js) and `Scan` (every **signed-in** user's report, including the raw CBOM JSON so Export Excel / Export JSON work on demand without re-scanning).
- **Guest scans** — not signed in? Scans still run for real, but nothing is written to the database. The report is returned to the browser and kept in `localStorage` (`src/lib/guest-scans.ts`), capped at `GUEST_SCAN_LIMIT` (2) and enforced server-side by an httpOnly cookie counter, not by the client-editable localStorage itself. The sidebar and `/scans` page read straight from that localStorage for guests.
- **Auth** — Auth.js (NextAuth v5), Google OAuth + email/password. Email/password signup requires clicking a real verification link sent from your own Gmail account (via SMTP) before the account can sign in — see "Email verification" below.

```
Browser → Next.js (Vercel) → Postgres/Neon (users, sessions, signed-in scans)
                            → Gmail SMTP (verification emails, your own account — no vendor)
                            → Scanner service (Railway, container) → target website

Guest scans never touch Postgres — they stay in the browser's localStorage.
```

### 1. Database (Neon)

1. Create a free project at [neon.tech](https://neon.tech).
2. Dashboard → your project → **Connect** → copy the connection string (the pooled one).
3. Put it in `.env` as `DATABASE_URL`.

```bash
cd web-platform
cp .env.example .env
npm install                            # also runs `prisma generate` via postinstall
npx prisma migrate dev --name init     # creates the User/Account/Session/Scan/VerificationToken tables on Neon
```

This project uses **Prisma 7**, which requires Node.js 20.19+ and a driver
adapter — `@prisma/adapter-neon` (see `src/lib/prisma.ts` and
`prisma.config.ts`). The generated client lives in `src/generated/prisma`
(gitignored, regenerated by `prisma generate`), not in `node_modules` —
that's new in v7, not a mistake.

**Nothing here installs globally.** `npm install` puts everything in
`web-platform/node_modules`; `npx prisma ...` runs the copy already in
`node_modules/.bin`, not a global install.

**Same Neon database works for both local dev and production** — no need for
a separate local database. If you'd rather keep them separate, create two
Neon projects (or two branches within one) and use a different
`DATABASE_URL` for each.

### 2. Auth secret

```bash
npx auth secret   # writes AUTH_SECRET into .env for you
```

If `.env` still has an **empty** `NEXTAUTH_SECRET=""` left over from
`.env.example`, delete that line — an empty string is treated as "no
secret" and is the single most common cause of Auth.js's generic "There was
a problem with the server configuration" error.

### 3. Email verification (Gmail SMTP — free, forever, no vendor account)

Email/password signup only works end-to-end once this is set — without it,
`/api/auth/register` will return a clear error instead of silently creating
an unverifiable account. This sends mail through **your own Gmail account**
over SMTP rather than a third-party email API — no signup elsewhere, no
plan, no free tier that could change later. A regular Gmail account can send
roughly 500 emails/day this way, far more than a student project needs.

1. Turn on 2-Step Verification on the Gmail account you want to send from: [myaccount.google.com/security](https://myaccount.google.com/security).
2. Google Account → **Security** → **App passwords** → create one, name it e.g. "PQShield" (search "App passwords" in the account settings search box if you don't see it directly — it only shows up once 2-Step Verification is on).
3. Put the Gmail address in `.env` as `GMAIL_USER`, and the 16-character app password (spaces don't matter) as `GMAIL_APP_PASSWORD`.

How it works (`src/lib/verification.ts`, `src/lib/email.ts`):

- Signup creates the user with `emailVerified: null` and emails a link containing a random token — the token is hashed (SHA-256) before being stored, so a database read alone can't be used to verify an account.
- The link hits `GET /api/auth/verify-email`, which checks the hash and expiry (24h), marks the account verified, and **deletes the token** so the link can't be reused.
- Credentials sign-in checks `emailVerified` and refuses (with a "resend verification email" option in the UI) until that link has been clicked.
- Google sign-in is auto-verified — Google already confirmed the email on their end.

### 4. Google OAuth (optional)

1. [Google Cloud Console](https://console.cloud.google.com/apis/credentials) → Create OAuth client (Web application).
2. Authorized redirect URIs: `http://localhost:3000/api/auth/callback/google` for local dev, plus your production domain's equivalent once deployed (both can be on the same OAuth client).
3. Put the client id/secret into `.env` as `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`.

If these are left blank, the Google button still renders but Google sign-in
is skipped from the provider list entirely (rather than crashing every
session check, which is what registering it with empty credentials does).

### 5. Scanner service (Python)

```bash
# from the repo root, one level above web-platform/
python3 -m venv .venv
source .venv/bin/activate      # .venv\Scripts\activate on Windows
pip install -r requirements.txt
uvicorn pqc_scanner.service:app --reload --port 8000
```

Leave this running in one terminal. `SCANNER_URL` in `.env` already defaults to `http://127.0.0.1:8000`, matching it.

### 6. Run the frontend

In another terminal:

```bash
cd web-platform
npm run dev
```

Open `http://localhost:3000`.

## Guests vs signed-in users

| | Guest | Signed in |
|---|---|---|
| Free scans | 2 per browser (httpOnly cookie counter) | Unlimited |
| Where scans live | `localStorage` only, never sent to the DB | Postgres/Neon — permanent, any device |
| Visible in sidebar / `/scans` | Yes, read straight from `localStorage` | Yes, queried from the DB |
| Export JSON/Excel | Yes — client sends its localStorage'd raw data to the export API directly (`POST`) | Yes — export API loads the raw data from the DB (`GET`) |
| Survives closing the tab | Yes (as long as localStorage isn't cleared) | Yes, and survives switching devices too |

The sidebar's collapse/expand toggle is plain in-memory component state —
intentionally **not** persisted anywhere, so it resets to expanded every time
the app loads.

## Demo report

The sidebar always shows exactly one bundled demo report (`demo.pqshield.io`,
id `pqshield-io-demo`) so the app never looks empty on first run. It's built
from fully synthetic scan data — no real organization's scan output is
bundled with the app. Its Export JSON / Export Excel buttons serve
`public/demo/cbom_demo.json` and `public/demo/audit_demo.xlsx` byte-for-byte
— those two files are the synthetic source and are never regenerated or
edited.

## Exporting reports

- **Export JSON** — downloads the raw CBOM scan data (same shape the CLI writes to `cbom_results.json`).
- **Export Excel** — downloads the same risk-report workbook the CLI's `-r/--report` flag produces, built by `pqc_scanner/risk_engine.py`.
- **Export PDF** — opens the browser print dialog with a print stylesheet that hides the sidebar/nav and expands every finding table.

## Deploying

1. **Scanner service** → push to Railway (or Fly/Render) using the root `Dockerfile`. Set `ALLOWED_ORIGINS` to your Vercel domain once you know it.
2. **Frontend** → push `web-platform/` to Vercel. Set the same env vars as `.env.example` in the Vercel project settings:
   - `DATABASE_URL` — your Neon connection string (same one works for prod; or use a separate Neon branch)
   - `AUTH_SECRET` — a different one than local dev, generate fresh with `npx auth secret`
   - `NEXTAUTH_URL` — your real Vercel domain, e.g. `https://pqshield.vercel.app`
   - `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` — same OAuth client as local dev works, as long as you added the prod redirect URI in Google Cloud Console
   - `GMAIL_USER` / `GMAIL_APP_PASSWORD` — same Gmail account works for both local dev and production, nothing to change at deploy time
   - `SCANNER_URL` — your deployed scanner service's URL (Railway/Fly/Render), not `127.0.0.1`
3. If you haven't already run `npx prisma migrate dev` against this exact `DATABASE_URL`, run `npx prisma migrate deploy` once (locally, pointed at the prod `DATABASE_URL`, or via Vercel's build command) to create the tables.
